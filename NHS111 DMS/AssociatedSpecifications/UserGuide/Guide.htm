User Guide

The Domain Message Specification (DMS) is the new style of message specification that replaces the old style Message Implementation Manual (MIM).

 

The main differences between a DMS and the MIM are:

1) A DMS covers a single domain, including one the artefacts required by that domain for implementation whereas the MIM covered all domains.

2) A DMS is not bound to either the SPINE or TMS but may be implemented with support for these as well as other mechanisms.

 

This section gives guidance on how to use a DMS and identifies the various artefacts and their purpose. Note: because the domain specifications vary slightly in structure, the purpose of all pages or artefacts described may not be present within all specifications.

1        Header and Footer

1.1        Header

 

 

The header contains the following information:

  • Domain name
    The business name of the domain, note there is no definitive definition of domain and this should be read as an area of interest or information flow.  
  • Version

    • T    he first number represents a major change, the second a minor change, although not definitive this should be interpreted as:
    • major = breaking changee
    • minor = non-breaking change.
  • Status
    • Draft - For information, evaluation and comment.  The message specification should not be used for implementation.
    • RC"n" -  (Release Candidate) A specification issued as fit for purpose but not yet proven through implementation.
    • Final - A full release of the specification which has been through the development life cycle and is deemed as fit for implementation. It will have been proven by a real implementation.   

       1.2     Footer

     

    • Document Ref
      A reference number for this Domain specification. 
    • Release Date
      The date of the release of the specification.
    • CFH Interoperability Team Contact Us
      Contact details for the interoperability Team which authored this specification. This address should be used only where other support contacts are not available.

2        Navigation Tabs  

2.1        Navigation Tab behaviour

When the mouse is over a tab the font colour changes and a screen tip appears.

 

   

When  viewing a page after clicking on a tab the font colour changes to indicate which page has been selected.

When navigating though multiple tab levels the font and tab colour changes to indicate which sub-page has been selected.

2.2        Navigation Tab Levels

The navigation tabs within a DMS can contain up to three levels, the following section describes the level one tabs. The purpose of the artefacts relevant to each tabbed page are described later.

 

Figure 1 below gives an example structure for a DMS.

 

Figure 1: Sample DMS Structure

  2.2.1      Level 1 Navigation Tabs

 

  1. Overview
    This takes you to the overview page which gives an overview of the domain
  2. Messages definitions
    This takes you to the definitions of the messages and maybe further divided into sub-tabs for some of the larger messaging domains.
  3. Associated Specification
    This page details artefacts that are generic in nature, this means they are used across many specifications and usually are related to common functionality.
  4. References
    This page provides details of artefacts which are used for reference and analysis.
  5. Glossary
    This provides a explanation of terms and acronyms used with the specification.

2.2.2        Level 2 Navigation Tabs for the Message Definitions Page

In most domain message specifications the level 2 navigation tabs detail the implementation specific information.

 

  1. TMS Implementation - This page details the information needed when using TMS as the transport layer for implementing the message flows.
  2. ITK Implementation - This page details the information needed when using the ITK distribution envelope over various transport layers when implementing the message flows. This can be web services, DTS or other layers.
  3. Configuration Profiles - This page gives information about versioning of configurable items.

  3. TMS Infrastructure: This link takes you to the 'TMS Infrastructure Domain Message Specification' which details information and artefacts used to transport message content over TMS.

3.2.1.4        Interaction Diagrams

The interaction diagram gives a simple visual overview of how the application roles exchange information with each other, a typical diagram is shown below in figure 9:

 

Figure 9: Sample Interaction Diagram

 

  1. The first application role - in this example this is the sending system.
  2. The second application role - in this example the receiving system.
  3. A description of the type of information flow.
  4. The interaction involved -  arrow gives direction of information flow, in this example information (a contact assessment) flows from left to right , from sending (originating) system to receiving system.
  5. The interaction involved -  arrow gives direction of information flow, in this example information (an acknowledgement) flows from right to left, from receiving system to (originating) system. It should be noted that both systems actually send and receive interactions. Most application roles are named based on the business information flow direction and things such as acknowledgements are not class as business information but as system responses.

Note: The interaction flows (the arrows) have no order or conformance. This means that interactions flow does not start at the top and go in the order downwards. Also the inclusion of an interaction does not mandate it.

Figure 10: Sample Interaction Table

3.3    Associated Specifications

This page is a list of generic specifications included within the DMS, these generic artefacts are used across many domains. The following sections detail what these are and there usage.

3.3.1    Generic CDA Page

3.3.1.1   Generic Models

The generic models are used across all domains to give consistency. These generic models are never implemented, but are used for level one validation. It is advised that vendors build in support for the generic model applicable to the domain to future proof any implementation. The domain models contained in the DMS will be a constraint of these generic models.

 

Figure 18: Generic Models Screenshot.

 

  1. Generic model name.

  2. Generic Model identifier.

  3. Link to the Generic R-MIM (model).

  4. Link to the tabular view  - there are no tabular views for generic models.

  5. Link to the message XML schema used for validation of message instances on the wire (Level 1).

  6. Link to the list of templates - Generic model do not use templates.

  7. Link to the list of reference templates - Generic model do not use references.

  8. A link to the MIF (XML representation of the message) - Note not all generic models have a MIF.
